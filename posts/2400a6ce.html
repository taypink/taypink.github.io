报文：

报文类型

DD报文: Database Description packet | LSDB：Link State Database

LSR报文：Link State Request packet | LSA：Link State Advertisement

LSU报文： Update packet

LSAck报文： Acknowledgment packet

报文类型 报文作用
Hello 周期性发送，用来发现和维持OSPF邻居关系。
DD 描述本地LSDB的摘要信息，用于两台设备进行数据库同步。
LSR 用于向对方请求所需的LSA。设备只有在OSPF邻居双方成功交换DD报文后才会向对方发出LSR报文。
LSU 用于向对方发送其所需要的LSA。
LSAck 用来对收到的LSA进行确认。

邻居关系建立

状态机

8种：

发现：Down $\rightarrow$ Attempt $\rightarrow$ Init $\rightarrow$ 2-way

领接：Exstart $\rightarrow$ Exchange

同步：Loading $\rightarrow$ Full

状态机 描述
Down 初始状态，没有收到任何 Hello 报文
Attempt 仅在 NBMA 网络中出现，主动向未回应的邻居发送 Hello 报文。
Init 收到Hello报文后，但对方的 Hello 报文里的“邻居列表”中还没有自己的 Router ID
2-way 收到的 Hello 报文中已包含自己的 Router ID；
非 DR/BDR 的 DROther 路由器之间会永远停留在该状态
此时“邻居关系”正式建立
Exstart 协商Master/Slave，确定DD的序列号
Exchange 交换DD报文，宣称自己的 LSDB 摘要目录
Loading 对比摘要后，向对方发送 LSR 请求缺少的 LSA，对方回应 LSU，本端回应 LSAck
Full 两端设备的 LSDB 完全同步
开始计算SPF

建立条件

  • Router-id 唯一（AS内）

  • 直连接口必须在同一网段

  • 网络类型一致

    P2P/Broadcast

  • MTU 一致 Check

  • Timer
    hello/dead

Check-list 报文载体 现象
Router-ID唯一（AS内） Hello / DD 直连冲突卡在 Init 或无法发现；
非直连冲突导致5类LSA震荡
同一网段（注意掩码） 接口层 / Hello 广播网下直接卡在 Down（互相不理会）
网络类型 接口层 / 逻辑 P2P 连 Broadcast 能到 Full，但两端计算出来的拓扑不一致，无法计算出路由！
MTU Check DD 报文 开启检查后，若不匹配会卡在 ExStart / Exchange
Timer (Hello/Dead) Hello 报文 Hello/Dead 间隔不一致，卡在 Down
Auth&PWD OSPF 头部 密码错或认证类型不匹配，直接丢弃报文，卡在 Down
区域属性 (Option) Hello 报文 E-bit（Stub属性）、N-bit（NSSA属性）不匹配，直接卡在 Down

报文认证

支持两种验证方式：

  • 区域验证
  • 接口验证（优先）

路由器类型

ASBR可以是任何OSPF路由器，且ASBR功能与ABR功能相独立，
OSPF域可以有ASBR而没有ABR，OSPF路由器可以同时是ASBR和ABR。

路由器类型 含义
IR（Internal Router） 该设备的所有端口都属于同一个 OSPF 区域，只维护所在区域的 LSDB。
ABR（Area Border Router） 用来连接骨干区域和非骨干区域，它会同时维护骨干区域和所连非骨干区域的 LSDB。它与骨干区域之间可以是物理连接，也可以是逻辑连接（如虚连接）。
BR（Backbone Router） 至少有一个接口属于骨干区域（Area 0）。所有的 ABR 和完全位于 Area 0 的内部设备（IR）都是骨干路由器。
ASBR（AS Boundary Router） 只要一台 OSPF 设备引入了外部路由，它就成为了 ASBR。
与其他 AS 交换路由信息的设备。ASBR 并不一定位于物理边界，它可能是区域内设备，也可能是 ABR。

六类LSA&外部路由

LSA类型

1/2类看区域内，3类过区域，4/5/7类向AS区外

Link State ID：它的标题写什么

Advertising Router：谁写的这封信

Type 名称 产生者 传播范围 Link State ID Advertising Router 核心内容
1 Router LSA ALL 仅本区域内 本路由器的 Router ID 本路由器的 Router ID 我是谁，我连着谁，我的开销是多少
2 Network LSA DR 仅本区域内 DR 的接口 IP 地址 DR 的 Router ID 这个 MA 网段都有谁连着我，掩码是多少
3 Summary LSA ABR 区域间 区域外路由的网段地址 通告该路由的 ABR 的 Router ID 我这儿有别的区域的路由：网段/掩码/开销
4 ASBR Summary ABR 除ASBR所在区域 ASBR 的 Router ID 通告该 ASBR 位置的 ABR 的 Router ID 去往 ASBR 该怎么走，开销是多少（为5类LSA导航）
5 AS External ASBR 整个 OSPF 域 外部路由的网段地址 ASBR 的 Router ID 外部引入的路由是多少（不属于任何区域）
7 NSSA LSA NSSA的ASBR 仅NSSA区域内 外部路由的网段地址 NSSA内 ASBR 的 Router ID 在 NSSA 区域内伪装成外部路由，在 ABR 处被转换成 5 类发往全网

Type 5 LSA 穿越Area 时候会不会变→区域外路由，ABR不会重新advertise，只需要flooding

Type 3/4 都是Area内的，穿越需要重新Advertise

Advertising Router、Metric会变更，State link ID不变

LSA 类型 ABR 的动作 Link State ID (内容) Advertising Router (谁发的) Metric (开销)
Type 3 (区域间路由) 重新通告 (Re-advertise) 不变 (目标网段 IP) 变 (变为当前 ABR 的 Router ID) 变 (累加 ABR 自己的开销)
Type 4 (找 ASBR 导航) 重新通告 (Re-advertise) 不变 (目标 ASBR 的 Router ID) 变 (变为当前 ABR 的 Router ID) 变 (累加 ABR 自己的开销)
Type 5 (外部路由) 原样泛洪 (Flooding) 不变 (外部网段 IP) 绝对不变 (永远是始发 ASBR 的 Router ID) 取决于类型 (Type 2不变，Type 1累加)

OSPF 路由聚合&过滤：

聚合

  • ABR聚合

    对Type 3 LSA聚合（针对区域内）

  • ASBR聚合

    对 Type5、Type7LSA聚合（针对外部引入路由）

    如果既是ASBR又是ABR，则对由Type7 LSA转化成的Type5 LSA进行聚合处理。

过滤：

可以使用的PBR：route-policy、access-list、prefix-list

不可在区域内过滤拓扑（LSA）： 由于 1/2 类 LSA 必须全区域完全一致，OSPF 无法在区域内阻止 LSA 的泛洪。你使用 filter-policy import 过滤路由，只能阻止路由进入当前路由器的路由表（RIB），但无法阻止 LSA 传递给下游邻居。下游路由器依然能学到路由！

区域间/边界过滤（Filter ABR/ASBR）： 只有在 ABR 传递 3类LSA，或者 ASBR 引入 5类LSA 的边界上，使用 filter 命令，才能真正将 LSA “拦腰斩断”，让下游路由器完全收不到。

OSPF区域与过滤

特殊区域 默认路由生成

区域类型 产生默认路由的 LSA 类型 产生条件/触发机制
STUB 3类 LSA 自动生成。只要配置了 Stub，ABR 就会无条件向区域内派发一条 3类的默认路由。
Totally STUB 3类 LSA 自动生成。
NSSA 7类 LSA 手工触发/不自动。在 ABR 上配置时，必须加上关键字（如华为：nssa default-route-advertise）才会产生 7类默认路由。
Totally NSSA 3类 LSA 自动生成。只要配置了 Totally，它的 ABR 就会自动产生一条 3类的默认路由。

为什么普通的 NSSA 区域 ABR 不自动生成默认路由？

→ 因为 NSSA 区域允许外来路由进来（有 ASBR），它本身可能就是个出口。如果 ABR 盲目自动生成默认路由引导流量去骨干网，可能会和 NSSA 区域原有的外部出口产生环路。所以 OSPF 把决定权交给了管理员。

NSSA区域：

NSSA（Not-So-Stubby Area）区域是OSPF特殊的区域类型。NSSA区域与STUB区域有许多相似的地方，两者都不传播来自OSPF网络其它区域的外部路由。差别在于STUB区域是不能引入外部路由，NSSA区域能够将自治域外部路由引入并传播到整个OSPF自治域中。

当区域配置为NSSA区域后，为保证到自治系统外的路由可达，NSSA区域的ABR将生成一条缺省路由，并发布给NSSA区域中的其他路由器。

配置NSSA区域时需要注意下列几点：

  • 骨干区域不能配置成NSSA区域。
  • 如果要将一个区域配置成NSSA区域，则该区域中的所有路由器都要配置NSSA区域属性。
  • 虚连接不能穿过NSSA区域。

Stub区域：

STUB区域是一些特定的区域，STUB区域的ABR不传播它们接收到的自治系统外部路由，在这些区域中路由器的路由表规模以及路由信息传递的数量都会大大减少。

STUB区域是一种可选的配置属性，但并不是每个区域都符合配置的条件。通常来说，STUB区域位于自治系统的边界，是那些只有一个ABR的非骨干区域。

为保证到自治系统外的路由依旧可达，该区域的ABR将生成一条缺省路由，并发布给STUB区域中的其他非ABR路由器。

配置STUB区域时需要注意下列几点：

  • 骨干区域不能配置成STUB区域。
  • 如果要将一个区域配置成STUB区域，则该区域中的所有路由器都要配置STUB区域属性。
  • STUB区域内不能存在ASBR，即自治系统外部的路由不能在本区域内传播。
  • 虚连接不能穿过STUB区域。
