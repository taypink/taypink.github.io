Reliability-HA

发表于2026-07-07|更新于2026-07-14|NetworkR&S
|字数总计:3.6k|阅读时长:11分钟|阅读量:

VRRP / BFD 负责“高可用与可靠性”，NQA / SNMP 负责“运维与监控”

常用组合：VRRP+BFD+NQA

  • VRRP 是“执行者” 负责搞出虚拟网关，实现主备备份。但它自己是个“近视眼”，只能感知自己接口的直连状态。如果隔壁运营商的骨干网断了，VRRP 根本不知道，流量还会傻傻地往主设备发。
  • BFD 和 NQA 是“眼线/联动工具”：
    • BFD（毫秒级）：负责死死盯着直连或者多跳的路由链路，断了立刻打小报告。
    • NQA（网络质量分析）：不仅能盯着断没断，还能盯着“堵不堵”（延时、抖动、丢包率）。
  • 联动效果：VRRP 绑定（Track）BFD 或 NQA。一旦眼线发现链路断了或质量太差，VRRP 立刻削减自己的优先级，备用路由器秒级/毫秒级顶上。

幕后大管家：SNMP（网络管理）

SNMP 的定位和前三者有些许不同，它是负责全局监控和收尸（告警）的。

  • 当 BFD 发现链路断开时……
  • 当 NQA 发现网络延迟超标时……
  • 当 VRRP 发生主备倒换，备用路由器变成主路由器时……
  • 它们都会第一时间给网络管理服务器（NMS）发送一个 SNMP Trap（告警报文）。网络管理员的监控面板上才会弹窗：“警告！XX链路断开，VRRP 已切换！”

VRRP

一、基础概念&版本差异

基础

  • 使用时只需配置\(Vrid\)和\(Virtual-IP\)，无需手动配置虚拟路由器的MAC地址

  • 优先级：默认100，越大越好

  • 虚拟MAC地址：Master 和 Backup 的虚拟 MAC 完全一致

    • v4:00-00-5E-00-01-[VRID的十六进制]

    • v6:00-00-5E-00-02-[VRID的十六进制]

流量行为分类(对内管理 vs 对外服务)

维度 对内管理维护(物理接口的对内维护流量) 对外提供服务(虚拟网关的对外业务流量)
IP类型 使用接口的实际物理 IP 虚拟 IP（VIP） 和虚拟 MAC
场景 路由器自己的 ARP 表里有一条“PC_A 对应某 MAC”的记录快到期了，路由器需要主动发个 ARP 探测去问一下：“PC_A 你还在不？在的话回个话，我刷新一下我的 ARP 表” 局域网里的 PC 发起 ARP 请求询问：“谁是网关 192.168.1.254，请告诉我你的 MAC 地址”

VRRP 组角色分类

说明
Normal 标准的常规 VRRP 备份组，独立运行
Administrator
(管理组 - mVRRP)		 负责发送心跳报文并绑定 BFD/NQA。它不直接承担业务流量，专职负责状态决定
Member
(业务成员组)		 通过直接绑定管理组(mVRRP)来同步状态
自身不再发送心跳报文，仅挂载虚拟 IP 转发业务流量，减轻CPU协议报文的处理压力
Load-Balance 在多网关架构中，使多台路由器同时处于活跃状态，分摊不同网段的流量
Load-Balance-Member
(负载分担成员)		 加入负载分担集群的具体数组成员

版本对比

v2 v3
组播地址 224.0.0.18 v4: 224.0.0.18
v6: FF02::12
通告间隔 默认1s ms级(最小为10ms)
故障切换速度 3秒以上 可实现毫秒级无感切换
认证机制 支持明文/MD5认证 协议本身取消了认证字段（交由底层安全机制处理）

\(Advertisement\)报文

VRRP只有这一种报文，使用组播，IP协议号为112

报文结构组成如下：

  • \(Ver\)：版本号
  • \(Type\)：固定为1；收到非 1 报文直接丢弃
  • \(Virtual~Rtr~ID\)：VRID，代表当前设备所属的备份组。
  • \(Priority\)
  • \(Count ~ IP ~ Addrs\)：表示这个虚拟路由器后面一共绑定了多少个虚拟 IP 地址
  • \(Auth ~ Type\)​：0：不认证、1：简单字符认证（明文密码）、2：MD5 认证
  • \(Adver~In\)：Advertisement Interval - 通告操作间隔，默认1 秒通告一次
  • \(Checksum\)

二、状态机和工作机制

状态机

状态 说明 选举/优先级
Initialize 此状态时设备不会对 VRRP 报文做任何处理 • 通常刚配置 VRRP 时或设备检测到故障时会进入 Initialize 状态
• 收到接口 Up 的消息后，若设备的优先级=255(即接口 IP 与虚拟 IP 相同)，则直接成为 Master 设备；如果设备的优先级小于 255，则会先切换至 Backup 状态
Master • 定时（Advertisement Interval）发送 VRRP 通告报文（组播方式）
• 以虚拟 MAC 地址响应对虚拟 IP 地址的 ARP 请求
• 转发目的 MAC 地址为虚拟 MAC 地址的 IP 报文
• 如果它是虚拟 IP 地址的拥有者，则接收目的 IP 地址为这个虚拟 IP 地址的 IP 报文		 • 如果报文的\(Priotity>自己\)，立即成为 Backup
• 如果报文的\(Priotity=自己\)，且\(\text{IP}_{\text{本地接口}}\)<\(\text{IP}_{\text{对端接口}}\)，立即成为 Backup
• 当接收到接口的 Shutdown 事件时，转为 Initialize 状态
Backup • 接收 Master 设备定时发送的 VRRP 通告报文，以此来监控 Master 的健康状态
• 不对对虚拟 IP 地址的 ARP 请求做任何响应
• 丢弃目的 MAC 地址为虚拟 MAC 地址的 IP 报文
• 丢弃目的 IP 地址为虚拟 IP 地址的 IP 报文		 • 如果 Master_Down 定时器超时（判定 Master 故障），或者收到了优先级比自己还低的通告报文（触发抢占），则转为 Master 状态
• 当接收到接口的 Shutdown 事件时，转为 Initialize 状态

自愿/主动ARP

当设备由 Backup → Master 切换成功，会主动发送Gratuitous ARP报文。 目的：刷新下挂设备的 MAC 地址表，使后续的业务流量能正确引导到新的 Master 路由器。

目的：主备倒换发生的瞬间，新 Master 主动大喊一声：“现在虚拟网关 IP 归我的物理接口管了！”，从而强制刷新局域网内所有交换机的MAC 地址表，使后续的业务流量能瞬间被正确引导到新的 Master 路由器，防止黑洞流量。

翻译成免费ARP的是真他妈傻逼

三、定时器

基础时间参数

  • 心跳间隔(Adver Interval)：默认 1秒。

  • 抢占延迟(Delay Time)：默认不限制（0秒），即发现自己优先级高就立刻抢占上位。

\(Master\_Down\)

故障判定时间

当Backup设备连续漏掉3个心跳包时，判定Master掉线。其总等待超时时间计算公式为：

\[Master\_Down = (3 \times \text{Adver\_Interval}) + Skew\_Time\]$

\(Skew\_Time\)

\[Skew\_Time = \frac{256 - \text{Priority}}{256}\]

  • 设计原因：假设局域网里有1台Master和2台Backup(Backup1 优先级 200，Backup2 优先级 100)。 当 Master 突然Down时，这两台 Backup 都会发现心跳停了。如果它们在同一时间发起抢占，网络就会发生冲突和震荡。

  • 解决方案：引入 \(Skew\_Time\)。优先级越高，计算出的 \(Skew\_Time\) 越短，从而让高优先级的备份设备优先上位。Backup1 的定时器会先超时，先一步升为 Master 并发送自愿 ARP，压制住低优先级的 Backup2。

BFD

会话建立方式

  • 静态：必须手动配置\(Local\)和\(Remote\)标识符。
  • 动态：自动协商

标识符Discriminator：\(A_{\text{Local}} = B_{\text{Remote}}\)

  • 只读：标识ID配置成功并激活后，只能通过重建会话来修改标识符ID
  • 唯一：本地设备的\(Local \neq Remote\)

端口与报文联动

BFD 协议跨越了传输层，控制报文与数据报文直接封装在 UDP 中，三个专属端口：3784、3785、4784

目的端口号3785为BFD echo报文使用，目的端口号3784与4784为BFD控制报文使用:3784为单跳检测报文，4784为多跳检测报文。

目的端口号 报文类型与应用场景 核心底层机制与网络行为
3784 控制面：单跳检测(Single-Hop) 直连防环强校验：运行于三层直连网段(同网段邻居)
触发GTSM机制(发送端将IP头部TTL强行固定为255。
接收端若收到\(TTL<255\)的报文，判定为绕路或伪造攻击，直接在硬件层静默丢弃
4784 控制面：多跳检测
(Multi-Hop)		 跨网段弹性转发：
用于跨越三层路由器的分布式链路（如跨网段 BGP 邻居、跨多跳静态路由）。解除了 TTL=255 的死锁限制，允许报文随三层寻路正常递减
3785 数据面：回声数据(Echo/Single-Arm Echo) 单边微秒级环回：专用于数据面快速检测
该报文的目的 IP 通常设置为“本地自身接口 IP”，目的 MAC 为对端。对端收到后无需 CPU 解封装，直接在 ASIC 芯片层线速转发环回，响应效率极高，适用于对端不支持 BFD 的场景

工作模式的区别

最常用异步

  • Echo必须和异步结合使用

异步模式：双方互相不停地发包（最常用）

按需/查询模式：轮询机制；没事不发包，有事（比如定时检查）才发包。

Echo功能：单向发包自己收，适用于对端不支持 BFD 的“单相思”场景。

单臂 Echo 功能虽然好用，但是它只支持网络层（IP）直连的情况，跨了三层设备就玩不转了

Mode 工作逻辑 公式
Asynchronous
(异步)		 被动等待对方的包，看对端DM
超出对端配置的DM个数，即认为链路中断		 \[\text{Time} = \text{协商后的 RX} \times \text{对端 DM} \]
Demand
(按需/查询)		 我说了算，看自己的DM
主动发包并要求对方回应		 \[\text{Time} = \text{协商后的 RX} \times \text{本地 DM} \]

Echo模式

本质：控制面和数据面分离

控制面端口 (UDP) 数据面端口 (UDP) 核心行为与标识符约束
标准 Echo 3784
(用于维持邻居)		 3785（用于数据面回声快速检测） 两端均配置 BFD。
控制面通过3784端口、异步模式维系邻居并交换标识符；
数据面通过3785端口互发echo报文实现高频检测。
Single-Arm Echo 无(不建立邻居) 3785（纯数据面流量打回） 对端不支持 BFD，仅需配置 Local discriminator
强制要求网络层(IP)直连，依赖对端底层ASIC芯片将报文环回。
单臂Echo
  • 只能应用在BFD单跳检测中，默认使用出接口IP为源IP
  • 只需要配置Local discriminator，无须配置Remote discriminator

实现：仅需一端支持 BFD(本地发包，对端不解封装、仅靠二/三层原生转发把包环回回来) →强制要求网络层(IP)直连（TTL=255，发出去转一圈回来 TTL=254）

⚠️避坑：由于存在以上特性，假如本地与对端设备之间隔了1台 L2 交换机，单臂 Echo 依然能通； 但如果中间隔了一台L3路由器，单臂Echo的网络层环回包会被中途的L3设备直接丢弃，导致会话无法建立

与其他协议联动

BFD 自己不产生路由，它是个“打小报告”的工具。所以它必须和别人绑定。你的笔记需要分门别类记录它的“朋友圈”：

  • BFD 联动静态路由：当 BFD 挂了，静态路由立刻在路由表中失效，浮动路由（备份）顶上。
  • BFD 联动动态路由（OSPF/BGP/ISIS）：
    • 大白话：OSPF 的 Hello 时间是 10 秒，死得太慢。绑定 BFD 后，毫秒级感知断开，通知 OSPF 立刻断开邻居，重新收敛。
  • BFD 联动 VRRP / Eth-Trunk：实现网关的秒级切换，或者捆绑链路中单根物理线条的故障感知。
  • 联动表项的本质：在笔记里画一个逻辑图，写明：物理链路故障 -> BFD会话Down -> 通知RM（路由管理） -> 路由表删线 -> 业务流量切换。

问题

1.本地看得到告警日志（Trap Buffer），但网管中心（NMS）收不到

→设备配置的SMNP版本与设备配置的Trap消息发送版本不一致

  • 生成 Trap(本地缓冲区)：这是设备自己内部的事情。只要设备发生了某种事件（比如接口 Down），它就会把这个事件记录到自己的内存缓冲区里（华为叫 display trap buffer，思科通常在 show logging 或 show snmp history 中可见）。这个过程不需要网管的参与。

  • 发送并解析 Trap(网管接收)：设备将这个事件打包成 UDP 报文发送给网管服务器。网管服务器收到后，必须用对应的凭证（版本、用户名、密码）去解密、校验。

    →如果设备和网管之间的安全凭证没对上，网管就会直接在后台静默丢弃这个报文，导致网管界面上什么都看不到。
