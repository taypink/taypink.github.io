MAC地址映射&冲突

32位只剩下28位可以自由组合，但是只买了23位给IP地址使用→有32个不同的IP地址对应一个MAC地址

(28位IP地址丢失头部5位)

组播 MAC 地址固定前缀： 前 24 位（高 24 位）固定为 OUI 01-00-5e，第 25 位被强制固定为 0

二进制解析
MAC冲突对应的IP计算

口诀：

  1. 后两段(第3、4字节)：直接照抄
  2. 第一段(第1字节)：把224~239 共16个地址写出来
  3. 第二段(第2字节)：看此处X数值大小，X>=128,那么X-128；如果X<128,那么X+128

于是:16(第一段的变化)*2(第二段的变化)=32个同一MAC地址的IP

例：找239.200.10.20的难兄难弟

  1. 后面两段地址照抄：x.x.10.20

  2. 16个地址：(224~239).x.10.20

  3. 因为\(200 \ge 128\),所以另外一个是\(200 - 128 = 72\)，第二段为200或72 于是冲突的IP：

    \(224.200.10.20 \longleftrightarrow 224.72.10.20\)

    \(225.200.10.20 \longleftrightarrow 225.72.10.20\)

    ……

    \(239.200.10.20 \longleftrightarrow 239.72.10.20\)

[思路梳理]：为什么口诀里第一段是 16 个、第二段是 \(\pm 128\)？

已知IPv4 组播地址总长 32 位，前四位固定位1110，只有28位可以分配，但是其中23位可用 那么我们不难发现→前4位固定，中间5位丢弃了，4+4=8刚好为第一段IP地址，第二节IP地址刚好被截断了1位！

  • 第二段 \(\pm 128\) 的秘密：第二段被截断了最高位（第 9 位）。这一位在十进制中代表 \(2^7 = 128\)。当这一位由 0 变 1 或由 1变 0 时，十进制数值刚好相差 128。
  • 第一段 16 个的秘密：第一段的后 4 位被丢弃了。4 位二进制可以有 \(2^4 = 16\) 种组合（0000 ~ 1111）。加上前 4 位固定的 1110（即 224），其范围自然就是 \(224 + 0\) 到 \(224 + 15\)，即 224 ~ 239。
  • 最终排列组合：第一段的 16 种变化 \(\times\) 第二段的 2 种变化 = 32 个冲突 IP。

  • IGMPv2 引入了离开报文 (0x17)，当最后一台收视主机离开时，会主动向 224.0.0.2（所有路由器）发声。查询器收到后立即连续发射 2 次 特定组查询，若 2 秒内没人应答，直接剪枝。这完成了从“百秒级”到“秒级”收敛的跨越。
  • 报告抑制机制的取消与接口状态重构（v2 ➔ v3）：
    • IGMPv1/v2 采用抑制机制，是为了防止网段内几百台 PC 同时发送 Report 撑爆路由器。但代价是路由器只知道“这个网段有人看”，不知道“到底是谁在看”和“想看哪个源”。
    • IGMPv3 为了支持 SSM，必须让路由器掌控每个主机的 (S, G) 绑定关系，因此取消了抑制机制。主机的 Report 报文不再针对单一组，而是打包成组记录(Group Record)，在一个报文中塞入多个组以及对应的 INCLUDE（仅接收指定源）或 EXCLUDE（拒绝指定源）的源列表
  • 向下兼容性底层死锁逻辑： 由网段内版本最低的设备决定。 如果接口配置了 IGMPv3，但有一台设备仅支持v2，整个接口的运行状态将强制降级切换回 IGMPv2。此时，该接口将自动丧失 SSM 特定源过滤能力，所有 v3 的特性将无法在该网段内生效

二层组播优化技术

IGMP Snooping 机制

配置该技术后，交换机可以侦听三层路由器与主机之间交互的 IGMP 报文，在二层建立精准的组播 MAC 转发表，从而将组播流量由纯粹的“二层广播泛洪”优化为“二层精准单播/组发”。→控制组播数据报文转发，抑制组播数据在二层网络中扩散

IGMP Snooping Proxy

在IGMPSnooping的基础上使交换机代替上游三层路由器，向下游主机发送IGMP Query报文、代替下游主机向上游设备发送IGMP Report和Leave报文，→节约上游设备和本设备之间的带宽。

报文处理归纳为“一留一丢”：

  1. 对于基础的“加入报告”和“普遍查询”报文，代理设备会选择处理或正常转发，用以维持和刷新基本的组播成员关系。
  2. 对于冗余的“特定组”和“特定源组”查询报文，它会选择直接丢弃以阻断其流向用户网段，从而达成精简报文数量、优化用户侧带宽的核心目的。
RP-路由器端口(Router Port)
  • 学习方式：通过动态接收 IGMP Query 普遍查询报文或 PIM Hello 报文自动生成
  • 老化机制：默认180/185s，如果在两个查询周期内未收到心跳，则直接老化删除该端口属性
MP-成员端口（Member Port)学习与老化
  • 学习方式：交换机通过监听下游主机回应的 IGMP 成员报告报文（Report）来自动记录
  • 老化机制：默认 260s(2 倍的 IGMP 查询间隔 + 最大响应时间，\(125\text{s} \times 2 + 10\text{s} = 260\text{s}\))华为→130s 若收到主机的 IGMP Leave 报文，则触发特定组查询，若 2s 内无响应则立即提前老化

IGMP Proxy

  • 现网中可能存在一台IGMP查询器需要管理大量组成员的情况，大量成员主机频繁加入/离开组播组时，会产 生大量的IGMP成员关系报告/离开报文，从而给IGMP查询器带来较大的处理压力。
  • 通过IGMP Proxy功能可减少IGMP查询器接收IGMP成员关系报告/离开报文的数量，减轻IGMP查询器压力。
  • IGMP Proxy通常被部署在IGMP查询器和成员主机之间的三层设备上。

RP&RPF

RP→Rendezvous Point

  • 1个RP可以为多个Group服务,但1个Group只能对应1个RP。
  • 适用协议范围：PIM-SM/ASM；
  • 组播源的流量(来自 FHR)需要通过单播注册寻找 RP，而接收者(来自 LHR)则通过向 RP 发送 Join 报文来构建共享树（RPT）
  • PIM-SM RPT是以RP为根，以存在组成员关系的PIM路由器为叶子

RPF-逆向路径转发检查

工作原理(防环核心)

思路：这个报文是从哪里来的，我该不该收”（来源）

  1. 检查源IP：当收到一个组播报文时，路由器会查看其 源IP
  2. 寻找出接口：路由器检查自己本地的单播路由表， 寻找“如果我要发报文给这个源 IP，我的出接口是哪个？”
  3. 对齐路径：收到组播报文的接口是否为单播路由表里的出接口(RPF)， 是就放行转发，不是就直接丢弃(视为环路)
选路优先级：

  1. 掩码最长

  2. 路单播路由协议的全局优先级（Preference / AD 值）

  3. 跨表最长前缀匹配优先。若掩码相同，则按“组播静态>MBGP>单播”的优先级

    路由类型 描述 选路优先级
    组播静态路由 (Multicast Static Route) 管理员手动配置的、专门为组播 RPF 检查服务的静态路由。 最高（通常默认管理距离/优先级最优）
    MBGP (Multicast BGP) 携带组播路由信息的扩展 BGP。 次高
    普通单播路由 (Unicast Route) OSPF、IS-IS、EIGRP、RIP 以及普通静态路由等。 默认兜底（看谁的协议优先级高就用谁）

接收端构建RPT树流程

控制面： \((*, G)\) 树 \(\rightarrow\) \((S, G)\) 树的两次逆向构建。RP 仅在前期充当流量中介，后期会被全网路由器“边缘化”。

数据面：FHR到RP的单播隧道封装➔单播与组播混合转发➔全线原生组播最优路径转发

1.RPT构建(共享树)→接收者入组

  • 主机向二层网络发送 IGMPv2 Report 报文，申请加入组G(此时主机并不知道源S)

  • LHR拦截了主机的Report报文，本地生成\((*, G)\) 表项；

    LHR查找去往RP的单播路由，从对应的RPF接口向RP的方向逆向发送 PIM Join 报文

  • 沿途的 Transit 路由器逐跳重复 RPF 检查，并在本地生成\((*, G)\) 表项，向源头转发 Join。 最终报文到达RP，逆向构建起一条以RP 为根、LHR为叶的RPT(共享树)

2.源端注册→组播源亮身份
  • 组播源Source发送组播数据，FHR接收到流量
  • FHR无条件将组播数据包封装进单播PIM Register报文发送给 RP，以此来通知 RP 组播源已激活。
  • RP 收到单播注册报文后，同时启动控制面和数据面的操作：
    • 数据面：剥离单播外壳，提取原生组播流，顺着第一阶段建立好的RPT共享树向下游的LHR泛洪；
    • 控制面：从报文中提取源地址S，主动向Source方向发送 \((S, G)\) Join报文，在RP和FHR间逆向构建Source-to-RP 的局部 SPT(源树)
  • 当组播数据顺着 SPT 正式到达 RP 后，RP 发现不再需要单播封装流量，随即向 FHR 发送单播 Register-Stop(注册停止) 报文。FHR 停止单播封装，控制面与数据面在 RP 处完成全面组播化
3.RPT向SPT切换→建立直连
  • LHR 收到组播流后，发现通过 RP 转发的路径属于次优路径(绕路)。LHR 立刻查找去往源\(S\)的单播路由表，从对应的 RPF 接口直接向源\(S\)的方向发射精准的 \((S,G)\) Join 报文。
  • 该 Join 报文在 LHR 与 FHR 之间沿途逆向建树，建立一条绕过 RP 的端到端最优路径 直连 SPT(源树)。
  • 此时，旧的 RPT 树和新的直连 SPT 树会同时向 LHR 发送重复的组播流量。
  • LHR 确认从直连 SPT 收到稳定的组播流后，立即向 RP 方向发送带有 RPT-bit(共享树标志位)的 \((S,G)\) Prune（剪枝）报文。RP 收到后，在自己的出接口列表(OIL)中剔除该 LHR。
  • 全网流量完全脱离 RP，组播流从源通过最优的直连 SPT 径直涌向接收者。RP 回归静默状态，仅维系控制面的表项握手。

SSM

  • 接收者精准知道\((S,G)\)（如现代 IPTV/数据中心复制）

  • 无需RP、RPT、注册组播源；可以直接组播源和组成员建立SPT

  • 周期性Join报文→组播分发树一直存在，不会因为没有组播流量而消失

  • SSM提前定义组播的源地址→可以在成员端DR上基于组播源地址直接反向建立SPT

  • 控制面只有\((S,G)\) 表项，天然防范组播 DOS 攻击

SSM Mapping

  • 目的：IGMP v1/v2报文无法携带组播源Source信息，使得IGMP v1/v2 也能使用v3 的SSM 服务
处理规则：

配置了SSM Mapping规则后，当IGMP查询器收到主机的IGMPv1或v2报告报文时，首先检查该报文中所携带的组播组地址G，然后根据检查结果的不同分别进行处理：

  • G在ASM范围呢，只提供ASM服务
  • G在SSM范围内(232.0.0.0/8):
    • 如果Router没有配置SSM Mapping→丢弃该报文
    • ​ 配置了→根据路由器配置的SSM映射规则，将IGMPv1/v2报告报文中的 \((*,G)\)信息转化为对应的\((G，INCLUDE，(S1,S2……))\)信息，以提供SSM组播服务。

ASM 和 SSM区别

特性 ASM (任意源) SSM (特定源)
表项 \((*, G)\) → 只匹配组地址 \((S, G)\) → 同时匹配源地址和组地址
组地址唯一性 全网唯一（同一时刻只能一个应用指用） 无需全网唯一（不同源可复用相同组地址）
地址利用率 低（浪费严重） 高（极大地节省了地址空间）
接收者控制力 弱（无法在加入时指定源，只能靠路由器过滤） 强（加入时主动指定接收哪个源的数据）
地址冲突后果 导致网络拥塞、接收端数据混乱 不会冲突，不同源的数据天然隔离

\((S, G)\) 路由表项用于在PIM网络中建立SPT。对于PIM-DM网络和PIM-SM网络适用。

\((*, G)\)路由表项用于在PIM网络中建立RPT。对于PIM-SM网络适用。

控制面&数据面→动态时序联动

PIM-SSM

  1. Host发起：PC通过IGMPv3发送Report报文，内容为：Include （1.1.1.1，232.1.1.1）（Source: 1.1.1.1, Group: 232.1.1.）
  2. LHR 拦截与翻译：
    • LHR 收到 Report 报文，在本地生成 (1.1.1.1, 232.1.1.1) 表项
    • LHR 提取源地址 1.1.1.1，查找本地路由表（单播路由/静态组播路由/MBGP），确定 RPF 上游接口
  3. 骨干网逆向建树：
    • 由于运行SSM，路由器无需查找RP，LHR直接从RPF接口朝元的方向发送PIM Join报文
    • 沿途路由器逐跳重复 RPF 检查并向源头转发 Join 报文，沿途全部生成(S，G)表项，逆向构建起一条直至 FHR 的 「SPT（源树）」
  4. 数据面转发&去冗余：
    • 顺向转发：组播源 Source 开始发送组播流量。流量顺着刚才建立好的 SPT 树状路径，无环、最优地顺向直达接收者。
    • 边界去冗余(Assert机制)：若多台路由器在同一个以太网网段同时向下游转发相同的组播流，触发 「PIM Assert（断言）」 机制，通过较量单播路由开销与接口 IP，强行锁定唯一转发者。

常见故障

⚠️二层组播流量不通

现象：未配置IGMPSnooping时，组播转发正常，配置了IGMPSnooping功能后，发现用户无法收到组播数据

  • 1.检查是否配置的IGMP Snooping Version较低。 如果配置的IGMP Snooping Version比用户主机的IGMP版本低，设备在收到IGMPReport报文后，只会 向路由器端口转发，不会生成成员端口和转发表项。

踩坑点 1：地址范围与服务模型不匹配**

  • 现象：配置了 PIM-SSM，但主机申请的地址在 224.x.x.x 或 239.x.x.x 范围内。
  • 根因：SSM 默认组范围为 232.0.0.0/8，非此范围的地址默认不触发 SSM 机制，导致 LHR 无法直接建立 (S, G) 表项。

⚠️ 踩坑点 2：二层与三层机制断节

  • 现象：控制面全通，但接收者收不到任何组播流。
  • 根因：交换机开启了 IGMP Snooping 但未配置「查询器（Querier）」，或者 LHR 接口未配置 ip pim 导致交换机无法学习到“路由器端口（Router Port）”，组播流量在二层被直接丢弃。

⚠️ 踩坑点 3：单播非对称路由引发的 RPF 死锁

  • 现象：单播物理链路全通，组播路由表项存在但 Incoming interface: Null。
  • 根因：单播路由表中去往组播源的接口，与组播报文实际流入的物理接口不一致（非对称路径），导致 RPF 检查失败。
  • 优化方案：必须强制使用「静态组播路由（Multicast Static Route）」或「MBGP」强行纠正组播的 RPF 选路。
